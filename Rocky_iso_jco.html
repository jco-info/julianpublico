Clúster HPC
con Rocky Linux

Construye desde cero un clúster de cuatro máquinas virtuales con autenticación centralizada LDAP, almacenamiento compartido NFS, gestor de trabajos Slurm y módulos de software dinámicos. Una infraestructura real de computación de alto rendimiento, a escala de laboratorio.

Rocky Linux 9 OpenLDAP + TLS NFS + Autofs Slurm WLM Environment Modules VirtualBox
Introducción

Descripción de la actividad

En el campo de la administración de sistemas y las infraestructuras de computación de alto rendimiento (HPC), es fundamental poder configurar entornos centralizados donde los usuarios puedan autenticarse desde cualquier nodo, compartir sus datos de forma transparente y ejecutar tareas de manera distribuida. Los centros de investigación, universidades y empresas tecnológicas gestionan cientos o miles de servidores como si fueran uno solo, gracias a tecnologías como LDAP, NFS, Slurm y Environment Modules.

🎯 Objetivos de la práctica
  • Diseñar e implementar una infraestructura de red interna con enrutamiento NAT
  • Configurar un servidor LDAP con TLS para autenticación centralizada
  • Configurar clientes LDAP con SSSD en los nodos del clúster
  • Implementar NFS y Autofs para directorios home compartidos
  • Desplegar Slurm como gestor de tareas con particiones diferenciadas
  • Instalar Environment Modules para gestión dinámica de software
  • Automatizar administración con SSH sin contraseña
⚙️ Requisitos previos
  • VirtualBox instalado en el equipo
  • ISO de Rocky Linux Minimal (versión 9)
  • Conocimientos básicos de Linux: edición de archivos, comandos de red, systemd
  • Al menos 8 GB de RAM disponibles para las cuatro VMs
  • ~15-20 GB de espacio en disco (clones enlazados)
Escenario

Una pequeña unidad de investigación necesita montar un clúster de computación con cuatro nodos en red interna 172.16.0.0/24, con salida a internet a través del nodo maestro mediante NAT.

Fase 1

Preparación del entorno

Esta fase es la base de todo el proyecto. Antes de instalar cualquier servicio, necesitamos que las cuatro máquinas virtuales estén correctamente configuradas, se vean entre sí y tengan acceso a internet a través del nodo maestro. Si esta fase no queda bien, todo lo que viene después fallará.

1.1Instalación de Rocky Linux Minimal

El punto de partida es tener Rocky Linux Minimal instalado en la máquina que actuará como nodo maestro (compute-0-0). Esta máquina es la más importante del clúster: será servidor LDAP, servidor NFS, controlador de Slurm y router para el resto de nodos.

ComponenteValor mínimo
Disco200 GB
RAM2 GB mínimo
Adaptador 1Red NAT (salida a internet)
Adaptador 2Red Interna (comunicación con nodos)
1.2Instalar nano

Rocky Linux Minimal no siempre incluye nano por defecto. Lo instalamos antes de empezar para poder editar archivos de configuración cómodamente a lo largo de todo el manual.

bash · compute-0-0
dnf install -y nano
1.3Configurar la red (NAT + Red Interna)

compute-0-0 necesita dos interfaces de red: una con salida a internet (NAT) y otra para comunicarse con los nodos del clúster (Red Interna). Primero comprobamos los nombres de las interfaces:

bash
ip a

En VirtualBox con Rocky Linux lo habitual es enp0s3 para NAT y enp0s8 para Red Interna. Configuramos con nmtui:

bash
nmtui
ℹ️

Dentro de nmtui: Edit a connection → enp0s3 (NAT): IP 10.4.104.110/8, Gateway 10.0.0.1, DNS 8.8.8.8. Para enp0s8 (Red Interna): IP 172.16.0.10/24, Gateway vacío.

bash · reiniciar interfaces
nmcli connection down enp0s3 && nmcli connection up enp0s3
nmcli connection down enp0s8 && nmcli connection up enp0s8
bash · verificar
ip a | grep inet
ip route
ping -c 2 8.8.8.8
✅ Resultado esperado
enp0s3: 10.4.104.110/8
enp0s8: 172.16.0.10/24
default via 10.0.0.1 dev enp0s3
1.4Configurar el hostname

El hostname identifica a cada máquina dentro del clúster. Slurm y otros servicios lo usan para reconocer los nodos, por eso es importante configurarlo correctamente.

bash
hostnamectl set-hostname compute-0-0
hostname
✅ Resultado esperado

1.5Configurar /etc/hosts

El archivo /etc/hosts permite que las máquinas se resuelvan entre sí por nombre sin necesidad de un servidor DNS. Añadimos todas las IPs del clúster.

1.6Configurar la contraseña de root

Establecemos una contraseña conocida para root, necesaria para las conexiones SSH iniciales entre nodos antes de configurar el acceso por clave.

bash
passwd root
# Contraseña: alumno (introducir dos veces)
1.7Configurar SSH para conexiones como root

Por defecto Rocky Linux no permite el login como root por SSH. Necesitamos habilitarlo para gestionar los nodos remotamente durante la configuración.

bash
nano /etc/ssh/sshd_config
líneas a modificar
PermitRootLogin yes
PasswordAuthentication yes
bash
systemctl restart sshd
systemctl status sshd
1.8Actualizar el sistema e instalar herramientas básicas
bash
dnf update -y
dnf install -y vim curl wget net-tools nmap telnet
1.9Deshabilitar el firewall

En este laboratorio deshabilitamos el firewall para evitar que bloquee el tráfico entre servicios como LDAP, NFS, Munge o Slurm.

⚠️

En un entorno de producción real esto no sería recomendable. Solo aplicable en entornos de laboratorio controlados.

bash
systemctl stop firewalld
systemctl disable firewalld
1.10Configurar el enrutamiento NAT

Este paso convierte a compute-0-0 en el router del clúster. Los nodos clientes no tienen conexión directa a internet, por lo que su tráfico pasará a través de compute-0-0.

bash · IP forwarding temporal
echo 1 > /proc/sys/net/ipv4/ip_forward
cat /proc/sys/net/ipv4/ip_forward
bash · IP forwarding permanente
nano /etc/sysctl.conf
# Añadir o descomentar:
# net.ipv4.ip_forward = 1

sysctl -p
bash · regla NAT con iptables
dnf install -y iptables iptables-services
iptables -t nat -A POSTROUTING -o enp0s3 -j MASQUERADE
service iptables save
systemctl enable iptables

Fase 2

Servidor LDAP en compute-0-0

LDAP (Lightweight Directory Access Protocol) es el servicio que permitirá que los usuarios julian1–julian4 puedan autenticarse en cualquier nodo del clúster con las mismas credenciales. Sin este servicio, cada máquina tendría sus propios usuarios locales y no habría forma de centralizar la gestión de accesos.

ℹ️

Todos los pasos de esta fase se ejecutan en compute-0-0 como root.

2.1Habilitar repositorio plus e instalar paquetes

Los paquetes de OpenLDAP para Rocky Linux se encuentran en el repositorio plus, que no viene habilitado por defecto.

bash
dnf update -y
dnf config-manager --set-enabled plus
dnf clean all
dnf install -y openldap-servers openldap-clients
bash · verificar instalación
rpm -qa | grep ldap
2.2Configurar la base de datos LDAP

OpenLDAP necesita un archivo DB_CONFIG que define parámetros de rendimiento como el tamaño de caché. Es importante que el propietario sea el usuario ldap.

bash
mkdir -p /var/lib/ldap

cat > /var/lib/ldap/DB_CONFIG <<EOF
set_cachesize 0 268435456 1
set_lg_regionmax 262144
set_lg_bsize 2097152
set_flags DB_LOG_AUTOREMOVE
EOF

chown ldap:ldap /var/lib/ldap/DB_CONFIG
chown ldap:ldap /var/lib/ldap
ls -la /var/lib/ldap/
2.3Iniciar y habilitar slapd
bash
systemctl enable --now slapd
systemctl status slapd
# Si hay errores: journalctl -xe | grep slapd
✅ Resultado esperado
active (running)
2.4–2.5Generar y aplicar contraseña del administrador

LDAP almacena las contraseñas en formato hash. El comando slappasswd genera ese hash a partir de la contraseña alumno.

⚠️

Guarda el hash generado. Lo necesitarás en varios pasos siguientes. Cada ejecución de slappasswd genera un hash diferente pero todos son válidos para la misma contraseña.

bash
slappasswd
# Introducir: alumno (dos veces)
# Ejemplo de resultado: {SSHA}FL7+NEcna8h5x96OtTrlEiPKOZSI+8Ar
bash
nano changerootpw.ldif
changerootpw.ldif
dn: olcDatabase={0}config,cn=config
changetype: modify
add: olcRootPW
olcRootPW: {SSHA}FL7+NEcna8h5x96OtTrlEiPKOZSI+8Ar
bash · aplicar
ldapadd -Y EXTERNAL -H ldapi:/// -f changerootpw.ldif
2.6Importar los esquemas básicos

Los esquemas definen qué tipos de objetos puede almacenar el directorio. Necesitamos cosine, nis e inetorgperson para crear usuarios del sistema.

bash

ldapadd -Y EXTERNAL -H ldapi:/// -f /etc/openldap/schema/nis.ldif
ldapadd -Y EXTERNAL -H ldapi:/// -f /etc/openldap/schema/inetorgperson.ldif

Fase 3

Configuración TLS en LDAP

Sin TLS, las comunicaciones LDAP viajan en texto plano. Cualquiera que intercepte el tráfico vería contraseñas y datos de usuarios. En esta fase creamos nuestra propia Autoridad Certificadora (CA), generamos un certificado para el servidor y configuramos LDAP para cifrar todas las conexiones.

ℹ️

Todos los pasos se ejecutan en compute-0-0 como root.

3.1Instalar openssl y crear estructura CA
bash
dnf install -y openssl
mkdir -p /etc/pki/CA
cd /etc/pki/CA
mkdir -p private certs newcerts crl
touch index.txt
echo 01 > serial
ls -la /etc/pki/CA/
3.2–3.4Generar clave privada y certificado de la CA

La clave privada de la CA es el elemento más sensible. Con ella se firman todos los certificados. Usamos 4096 bits y permisos 600.

bash
cd /etc/pki/CA
openssl genrsa -out ca.key 4096
chmod 600 ca.key
openssl req -new -x509 -days 365 -key ca.key -out ca.cert.pem

3.5Crear extensiones SAN

Las extensiones SAN (Subject Alternative Names) son necesarias para que los clientes modernos acepten el certificado. Sin ellas los clientes LDAP rechazarían la conexión TLS.

bash
nano /etc/pki/CA/server_cert_ext.cnf
server_cert_ext.cnf
[v3_ca]
basicConstraints = CA:FALSE
nsCertType = server
nsComment = "OpenSSL Generated Server Certificate"
subjectKeyIdentifier = hash
authorityKeyIdentifier = keyid,issuer:always
keyUsage = critical, digitalSignature, keyEncipherment
extendedKeyUsage = serverAuth
subjectAltName = @alt_names

[alt_names]
IP.1 = 172.16.0.10
IP.2 = 172.16.0.11
IP.3 = 172.16.0.12
IP.4 = 172.16.0.13
IP.5 = 172.16.0.14
DNS.1 = ldap.julian.local
3.6–3.8Generar clave del servidor y CSR
bash
mkdir -p /etc/pki/CA/private
cd /etc/pki/CA/private
openssl genrsa -out ldap.julian.local.key 4096
chmod 600 ldap.julian.local.key
openssl req -new -key ldap.julian.local.key -out ldap.julian.local.csr
# Completar con los mismos datos que el certificado CA
# Challenge password: (dejar vacío)
3.9Firmar el certificado del servidor con la CA
bash
cd /etc/pki/CA
openssl ca -keyfile ca.key -cert ca.cert.pem \
  -in private/ldap.julian.local.csr \
  -out private/ldap.julian.local.crt \
  -extensions v3_ca -extfile server_cert_ext.cnf
# Responder "y" a "Sign the certificate?" y "Commit?"
bash · verificar
openssl verify -CAfile ca.cert.pem private/ldap.julian.local.crt
openssl x509 -noout -text -in private/ldap.julian.local.crt | grep -A 1 "Subject Alternative Name"
✅ Resultado esperado
private/ldap.julian.local.crt: OK
3.10Copiar certificados y aplicar TLS en LDAP
bash · copiar certificados
mkdir -p /etc/openldap/cacerts
cp /etc/pki/CA/private/ldap.julian.local.crt /etc/openldap/certs/
cp /etc/pki/CA/private/ldap.julian.local.key /etc/openldap/certs/
cp /etc/pki/CA/ca.cert.pem /etc/openldap/cacerts/
chown -R ldap:ldap /etc/openldap/certs
chown -R ldap:ldap /etc/openldap/cacerts
chmod 600 /etc/openldap/certs/ldap.julian.local.key
bash · aplicar TLS (tls7.ldif)
nano /root/tls7.ldif
tls7.ldif
dn: cn=config
changetype: modify
replace: olcTLSCertificateFile
olcTLSCertificateFile: /etc/openldap/certs/ldap.julian.local.crt
-
replace: olcTLSCertificateKeyFile
olcTLSCertificateKeyFile: /etc/openldap/certs/ldap.julian.local.key
bash
ldapmodify -Y EXTERNAL -H ldapi:/// -f /root/tls7.ldif
# Añadir también el certificado CA (tls7_1.ldif):
nano /root/tls7_1.ldif
tls7_1.ldif
dn: cn=config
changetype: modify
add: olcTLSCACertificateFile
olcTLSCACertificateFile: /etc/openldap/cacerts/ca.cert.pem
bash
ldapmodify -Y EXTERNAL -H ldapi:/// -f /root/tls7_1.ldif
3.11Configurar cliente local y verificar TLS
bash
nano /etc/openldap/ldap.conf
# Añadir al final:

# TLS_CACERT /etc/openldap/cacerts/ca.cert.pem

systemctl restart slapd
